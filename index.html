SEMVER

Semantic Versioning

Platform independent versioning to prevent dependency hell.

Not only used by Composer, but that's our usecase.


Given a version number MAJOR.MINOR.PATCH, increment the:

    MAJOR version when you make incompatible API changes,
    MINOR version when you add functionality in a backwards-compatible manner, and
    PATCH version when you make backwards-compatible bug fixes.

Additional labels for pre-release and build metadata 
are available as extensions to the MAJOR.MINOR.PATCH format.

Given a version number MAJOR.MINOR.PATCH

1.2.3

Change the MAJOR version when you make incompatible API changes

What is an API?

(in this context)

Anything accessible to other developers

  • Public Methods
  • Method Signatures
  • Return Types
  • Data Structures

Change the MINOR version when you add functionality in a backwards-compatible manner

This is an API change!

But old code still works.

Usually new methods, classes etc.

Deprecate methods and classes with @deprecated

Change the PATCH version when you make backwards-compatible bug fixes

Only change unexpected behavior to expected behavior

No API changes at all allowed!

Magento has a different definition of API.

Service Contracts

AKA The API Directory

AKA Interfaces

Why?

  • API is old, and will change frequently
  • API is too big
  • People need modules that won't break unexpectedly

A Service Contract is a deal between magento and 3rd party developers.

They will let us know if anything in the contract changes.

They will try to keep changes to the contract to a minimum.

More info? It's in the docs!

To summarise:

Magento will change a module's Major Version if it's Service Contracts change.

Magento will change a module's Minor Version if it adds to it's Service Contracts.

Magento will change the patch version if it's Service Contracts stay the same.
